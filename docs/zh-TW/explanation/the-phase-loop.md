# 階段迴圈

> GSD Core 組織工作的核心思維模型。

---

## 迴圈是什麼

GSD Core 將所有開發工作結構化為一個重複週期：

```text
Discuss → (UI design) → Plan → Execute → Verify → Ship
```

每個工作單元——稱為**階段**——按順序經歷這些步驟。迴圈並非形式主義。每個步驟的存在都是為了防範某一類特定的失敗，而這類失敗是上一步驟單獨無法預防的。

本文件解釋*為什麼*迴圈是這種形態。有關執行每個步驟的操作說明，請參見底部連結的操作指南。

---

## 每個步驟存在的原因

### Discuss

在知道*如何*構建某物之前，規劃無法開始——而不僅僅是*構建什麼*。`ROADMAP.md` 中的階段目標描述了結果。Discuss 步驟捕捉塑造通往該結果路徑的實現決策：選用哪些庫、採用哪種錯誤處理策略、某功能是按路由還是全域性實現、邊緣情況應如何處理。

沒有 Discuss 步驟，規劃器必須自行做出這些判斷。有時它猜對了。但往往猜得似是而非卻實際有誤——產出一個邏輯連貫卻與你實際偏好相悖的計劃。等到執行完成、發現錯誤時，你已經在撤銷大量工作了。

Discuss 步驟刻意保持輕量。它是一次對話，而非規格說明練習。輸出是階段目錄中的 `CONTEXT.md`：一份規劃器、執行器和驗證器都可以閱讀的結構化決策記錄。對話只需幾分鐘，卻能節省數小時的返工時間。

### UI design（可選）

對於有視覺元件的階段，在 Discuss 和 Plan 之間有一個可選的 `/gsd-ui-phase` 步驟。它生成 `UI-SPEC.md`——一份在編寫任何程式碼之前描述佈局、互動和視覺行為的設計契約。當 UI 足夠複雜，設計中的歧義會導致不同的實現選擇時，值得執行此步驟。一份清晰的設計契約比重新實現要便宜得多。

### Plan

Plan 步驟完成執行所需的研究、分解和結構性思考。它以一系列全新上下文的子代理執行：一個研究者調查生態系統並將發現記錄在 `RESEARCH.md` 中，一個規劃器同時閱讀研究結果和 `CONTEXT.md` 以生成 `PLAN.md` 檔案，以及一個計劃檢查器驗證計劃是否完整、一致且在範圍內。

計劃包含什麼？每個 `PLAN.md` 描述一個有邊界的工作單元：需要修改的檔案、要做的具體更改、定義完成狀態的驗收標準。計劃按依賴波次排序，以便並行執行是安全的——同一波次中的執行器處理互不重疊的關注點。

Plan 步驟是歧義代價最高的時刻。一個模糊的計劃會產生一個需要做假設的執行器。多個並行執行器對同一關注點做出不同假設會產生衝突。計劃檢查器的工作是在執行開始前捕捉這些問題，而不是事後。

### Execute

執行執行計劃。每個執行器獲得一個全新的 200k token 上下文視窗，其中精確載入了它所需的內容：專案摘要、階段上下文、研究結果，以及其任務對應的特定 `PLAN.md`。僅此而已。

執行器編寫程式碼並原子性地提交。每次提交對應計劃中的一個已完成任務。當一波並行執行器完成時，協調器合併其狀態並開始下一波。

執行器的全新上下文不是便利設施——它是防止上下文腐化的機制。一個帶著 180k token 積累會話歷史執行的執行器是一個性能退化的執行器。一個乾淨啟動、只讀取其計劃所需內容的執行器，是以滿狀態執行的執行器。

### Verify

所有執行器完成後，一個驗證器代理讀取階段目標、`CONTEXT.md` 決策、計劃和執行摘要——並檢查構建的內容是否與預期相符。它生成 `VERIFICATION.md`，如果存在差異，則生成有針對性的修復計劃。

驗證不僅僅是測試。它檢查需求覆蓋率（所有 REQ-ID 都被處理了嗎？）、決策覆蓋率（`CONTEXT.md` 中記錄的決策是否實際實現了？），以及整體階段目標對齊情況。階段完成不是因為執行沒有報錯。而是因為構建的內容符合計劃，計劃的內容符合決策。

### Ship

Ship 步驟建立拉取請求並歸檔階段產出物。`STATE.md` 更新以標記階段完成。迴圈隨後為下一個階段重新開始。

## 里程碑與階段

**里程碑**是一個版本週期——專案有意義的、可釋出的增量。它有名稱、版本號，以及定義其必須交付內容的一組需求。當里程碑的所有階段都已釋出且需求都已覆蓋時，里程碑才算完成。

**階段**是里程碑中的一個工作單元。階段有目標、它所處理的一組需求，以及實現它的一組計劃。

這種關係很重要，因為里程碑和階段有不同的關注範圍。里程碑問："這個版本的產品能做什麼，不能做什麼？"階段問："下一個我們可以研究、規劃、執行和驗證的有邊界的事情是什麼？"

里程碑邊界劃定在自然產品邊界處——一個可部署的 API、一個可用的 UI 流程、一個完整的資料模型。階段邊界劃定在可以在一次迴圈中安全執行而不使迴圈變得笨重的工作量極限處。

## 什麼是好的階段範圍

這值得深入探討，因為它是迴圈中最常見的摩擦來源。

階段太大會變成一個獨立的研究專案。規劃器難以將其分解為獨立計劃。後續波次的執行器被阻塞，等待前面的波次。驗證變成全面審計而非有針對性的審查。反饋週期從數小時延伸到數天，而在大量程式碼編寫後期才發現根本性設計錯誤的風險急劇上升。

階段太小則會將本屬於一體的工作碎片化。你最終會得到只有幾行的計劃檔案、在幾分鐘內完成的階段，以及規劃開銷遠超執行成本的局面。迴圈感覺像官僚主義而非幫助。

好的階段範圍具備以下特徵：

- 目標可以用一句話表述，既不顯然瑣碎，也不可疑寬泛。
- 規劃所需的研究是有邊界的——生態系統問題有答案，且不依賴於其他階段的先行完成。
- 執行可以並行化為少數幾個不重疊的計劃，而不是幾十個。
- 有清晰的、可測試的完成定義，驗證器無需閱讀整個程式碼庫即可檢查。

具體來說："新增 HMAC-SHA256 簽名驗證中介軟體"是一個好的階段範圍。"構建認證系統"通常不是——它幾乎總是包含多個獨立關注點，作為單獨階段會更好。"修復 README 中的拼寫錯誤"低於迴圈能增添價值的門檻；請使用 `/gsd-quick` 代替。

有疑問時，拆分。更小的階段完成得更快，驗證更有把握，當設計決策被證明有誤時也更容易糾偏。

## `.planning/` 如何在迴圈中保持狀態

迴圈不是單次會話。研究、規劃和執行可能跨越多個會話，中間有上下文重置。`.planning/` 目錄使這成為可能。

迴圈的每個步驟都讀取早期步驟產生的產出物，併為後續步驟寫入產出物。Discuss 步驟生成的 CONTEXT.md 在規劃器執行時仍然可用——即使那是數小時後的不同會話。規劃器生成的 PLAN.md 檔案在執行器執行時仍然可用——即使跨越重啟。驗證器寫入的 VERIFICATION.md 在你審查階段時仍然可用。

`STATE.md` 是凌駕於這一切之上的導航層。它精確記錄專案當前在迴圈中所處的位置：哪個里程碑處於活動狀態，哪個階段正在進行，哪些計劃已完成，哪些待處理。任何需要定向的代理或工作流都首先讀取 `STATE.md`。

有關這些檔案的精確結構，請參見[規劃產出物](../reference/planning-artifacts.md)和 [STATE.md 架構](../reference/state-md.md)。

## 迴圈是一種節奏，而非約束

很容易將迴圈視為官僚主義——一組你在被允許編寫程式碼之前必須執行的步驟。這種框架是錯誤的。

迴圈之所以存在，是因為每個步驟都能防範在事後修復真正代價高昂的失敗。Discuss 防止在錯誤假設上規劃。Plan 防止執行從根本上存在缺陷的設計。Verify 防止釋出偏離了需求的工作。這些不是人為製造的問題。它們是真實功能規模的 AI 輔助開發的實際失敗模式。

迴圈執行良好時，感覺像是一種節奏：有節奏的專注、有邊界的工作，每個步驟都清晰，因為上一步驟做好了它的工作。開銷是真實的，但它是前置支付的——用幾分鐘規劃換取而不是數小時返工。

對於低於迴圈門檻的工作，GSD Core 提供更輕量的原語。階段迴圈是一種工具，而非唯一工具。
