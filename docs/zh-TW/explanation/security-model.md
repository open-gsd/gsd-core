# GSD Core 安全模型

> **說明** — 本文件描述 GSD Core 為何採用當前的安全立場，以及各防護層如何協同工作。本文不是每個鉤子引數的參考手冊。有關 `/gsd-secure-phase` 命令及其選項，請參閱[命令文件](../COMMANDS.md)。有關實現層面的鉤子架構，請參閱[架構文件 § 鉤子系統](../ARCHITECTURE.md#hook-system)。有關組織級安全基線（掃描器控制、事件處理清單、所有權模型），請參閱 [SECURITY.md](../../../SECURITY.md)。

---

## 為何 AI 驅動的開發需要專項安全立場

傳統程式碼編輯器不會代表使用者執行任意軟體包。GSD Core 會。其研究 → 計劃 → 執行的流水線將從"命名一個軟體包"到"執行 `npm install <package>`"、從"編寫規劃產物"到"將該產物用作 LLM 系統提示"的完整路徑全部自動化。每一個自動化步驟都將人從環路中移除——而每次移除都是潛在的攻擊面。

GSD Core 的安全模型圍繞一個核心原則構建：**縱深防禦**。沒有任何單一控制措施被認為是完美的。多個相互重疊的層各自降低一類特定風險，共同使攻擊面大幅難以利用——儘管並未徹底消除。本文件末尾的誠實總結說明了該系統無法防禦的內容。

## 第一層 — 供應鏈保護：軟體包合法性門控

### 威脅

AI 模型會產生幻覺性軟體包名稱。這並非邊緣故障模式：2025 年的研究表明，AI 生成的軟體包引用中約有 20% 是幻覺名稱，與合法軟體包並不對應。這些幻覺名稱中有一部分——同一研究中約 43%——在提示詞中持續重複出現，這意味著攻擊者可以觀察 AI 工具常見生成的名稱，然後在 npm、PyPI 或 crates.io 上預先註冊這些名稱，並附帶惡意的安裝後腳本。這種技術稱為 *slopsquatting*。

Slopsquatting 的隱蔽之處在於，通過 `npm view` 驗證的幻覺名稱*看起來是合法的*。登錄檔條目僅證明有人註冊了該名稱——並不能證明該軟體包實現了 AI 所描述的功能，也不能證明它有任何合法使用者，更不能證明其安裝指令碼是安全的。若沒有門控，幻覺名稱將無聲地流過 GSD 的研究員 → 規劃員 → 執行員流水線，最終在您的機器上作為 `npm install <attacker-package>` 執行。

### 門控機制

門控機制跨三個流水線階段執行：

**研究階段。** 當 `gsd-phase-researcher` 推薦外部軟體包時，它會對每個軟體包執行 `slopcheck install <pkgs> --json`。結果會以 `## Package Legitimacy Audit` 表格的形式寫入 `RESEARCH.md`。標記為 `[SLOP]`（高置信度幻覺或攻擊者註冊）的軟體包在儲存前會**從 `RESEARCH.md` 中完全刪除**，永遠不會到達規劃員。

**規劃階段。** `gsd-planner` 讀取審計表。對於任何標記為 `[SUS]`（可疑：新註冊、下載量低、無原始碼倉庫，或命名模式接近某熱門軟體包）或 `[ASSUMED]`（來自 WebSearch 而非直接登錄檔驗證）的軟體包，規劃員會在安裝步驟之前**插入一個 `checkpoint:human-verify` 任務**。該檢查點包含指向登錄檔頁面的直接連結，以及需要重點核查的內容：維護者歷史、問題跟蹤器活動、是否存在可疑的安裝指令碼。

**執行階段。** 若安裝失敗，`gsd-executor` 會**觸發檢查點並停止**。它不會靜默地嘗試備用軟體包名稱——因為備用名稱本身可能也是惡意的。這是執行員行為定義中的明確規則（執行員代理定義中的 RULE 3）。

### 為何 WebSearch 軟體包始終標記為 `[ASSUMED]`

通過 WebSearch 發現的軟體包名稱無論 `npm view` 是否成功，均標記為 `[ASSUMED]`。在登錄檔中存在的軟體包不等於可以安全安裝的軟體包。`npm view` 僅證明註冊，而非合法性。`[ASSUMED]` 標籤與 `[SUS]` 觸發相同的人工驗證檢查點，確保任何未經驗證的網路發現推薦在安裝前均須經過人工稽核。

### 生態系統覆蓋範圍

研究員使用各生態系統專屬的驗證命令，而非單一通用檢查：

- Node.js：`npm view`
- Python：`pip index versions`
- Rust：`cargo search`

這覆蓋了跨生態系統幻覺——根據 2025 年 USENIX 研究，其發生率約為 9%——即 AI 推薦的軟體包存在於某個生態系統，但並不存在於實際使用的生態系統中。

### 優雅降級

若 `slopcheck` 不可用（未安裝，或研究階段 pip 安裝失敗），GSD 應用最嚴格的兜底策略：**每個推薦軟體包均標記為 `[ASSUMED]`**，規劃員對每次安裝均設定 `checkpoint:human-verify` 任務。研究和規劃照常進行——系統不會因缺少工具依賴而硬性失敗。這有意比正常流程更嚴格：`slopcheck` 不可用意味著每次軟體包安裝都會有人工檢查點。

`slopcheck` 工具採用 MIT 協議，可通過 pip 安裝。若該工具被廢棄，`[ASSUMED]` 門控兜底策略確保人工檢查點覆蓋無論如何均能維持。

## 第二層 — 提示注入防禦

### 威脅

GSD Core 生成的 Markdown 檔案會成為 LLM 系統提示。研究流水線讀取外部網頁內容；規劃流水線接受使用者提供的文本（`--text-file`、`--prd`）；執行流水線寫入規劃產物，這些產物稍後會作為代理上下文被重新讀取。任何流入這些產物的使用者可控文本都是潛在的**間接提示注入**向量——攻擊者控制的字串一旦進入系統提示，就會嘗試覆蓋代理指令或竊取資訊。

### 防禦機制

GSD Core 在三個層面應對提示注入。

**輸入驗證（`security.cjs`）。** `get-shit-done/bin/lib/security.cjs` 模組是核心安全工具。它提供：

- 路徑遍歷防護：使用者提供的檔案路徑（`--text-file`、`--prd`）經過驗證，確保解析在專案目錄內，並顯式處理 macOS `/var` → `/private/var` 符號連結解析
- 提示注入檢測：已知注入模式（角色覆蓋、指令繞過、系統標籤注入）在使用者提供的文本進入任何規劃產物之前進行掃描
- 安全 JSON 解析：防止通過精心構造的 JSON 負載發動原型汙染攻擊的包裝器
- Shell 引數驗證：傳遞給子 Shell 命令的引數在使用前經過驗證

**執行時鉤子：`gsd-prompt-guard.js`。** 該鉤子在每次針對 `.planning/` 檔案的 Write 或 Edit 呼叫時觸發。它掃描待寫入內容中的注入模式，與 `security.cjs` 相同（部分模式直接內聯到鉤子中以實現獨立性——鉤子不 `require()` 該模組，因此即使模組路徑改變也能執行）。檢測結果**僅供參考**：鉤子記錄發現但不阻止寫入。其原因在於，對合法規劃寫入的誤報攔截比二級掃描層漏掉一次注入更具破壞性。

**執行時鉤子：`gsd-read-injection-scanner.js`。** 該鉤子在每次 Read 工具呼叫的輸出時觸發。它掃描*剛剛讀取的內容*中在不可信內容中注入的指令——捕獲攻擊者在 GSD 即將納入代理上下文的檔案中嵌入指令的情況。

**CI 掃描器。** `prompt-injection-scan.test.cjs` 作為測試套件的一部分，掃描所有代理、工作流和命令檔案中嵌入的注入向量。這能捕獲 GSD 原始碼本身的注入嘗試——例如，修改工作流檔案以新增角色覆蓋指令的供應鏈攻擊。

### 讀取注入掃描器與提示守衛的對比

兩個鉤子覆蓋互補的攻擊面。`gsd-prompt-guard.js` 監視*對規劃產物的寫入*——捕獲注入的植入。`gsd-read-injection-scanner.js` 監視*任何檔案的讀取*——捕獲來自外部內容的注入攝取（依賴項的 README、第三方配置檔案、使用者提供的文件）。兩者共同覆蓋了攝取 → 儲存 → 再讀取的完整生命週期。

## 第三層 — 倉庫與依賴完整性

在 GSD 執行時行為的上游，`open-gsd` 組織在倉庫和軟體包層面實施控制。這些控制在 [`docs/security/baseline.md`](../../security/baseline.md) 中有完整文件，此處為完整性摘要。

**依賴完整性。** 所有第三方依賴通過 `package-lock.json` 鎖定，並在安裝前與已釋出的校驗和進行驗證。`scripts/check-npm-integrity.cjs` 門控在 CI 階段檢測版本無效、缺失軟體包和多餘軟體包。這能緩解針對 GSD 自身依賴的依賴混淆和拼寫搶注攻擊。

**金鑰掃描。** 每次提交和 PR 均掃描硬編碼金鑰。有意的測試夾具必須使用專案標準排除語法進行標註（標註格式見 `SECURITY.md`）。未標註的抑制項將導致 CI 失敗。

**區域設定安全文本掃描。** 輸出和麵向用戶的字串會掃描 Unicode 同形字元、雙向覆蓋字元以及不可見 Unicode——即 CVE-2021-42574（"特洛伊木馬原始碼"）中記錄的那類可在差異中隱藏惡意內容的攻擊。

## 權衡與侷限

本文描述的安全模型切實降低了 AI 驅動開發的攻擊面，但並未消除供應鏈風險。

**軟體包合法性門控降低的風險：** 幻覺或攻擊者註冊的軟體包在沒有人工檢查點的情況下到達 `npm install` 的機率。`[SLOP]` 門控徹底刪除高置信度的惡意軟體包；`[SUS]` / `[ASSUMED]` 門控要求在執行前進行人工稽核。這大幅提升了成功 slopsquatting 攻擊的成本。

**軟體包合法性門控無法消除的風險：** 之後遭到入侵的合法軟體包（賬戶接管、其自身依賴樹中的依賴混淆）不會被 slopcheck 捕獲——slopcheck 在研究階段檢查註冊訊號。鎖定檔案和依賴完整性層的 `npm audit` 才是應對此類攻擊的控制手段。

**提示注入防禦降低的風險：** 規劃產物中使用者可控文本成功覆蓋代理指令的機率。基於模式匹配的已知注入形式能捕獲常見情況；新穎的越獄方法或低訊號注入可能無法被檢測到。僅供參考的立場意味著檢測結果被記錄但不會被阻止——這是一個經過深思熟慮的選擇，以犧牲在檢測時硬性停止為代價換取工作流的連續性。

**提示注入防禦無法消除的風險：** 足夠有創意的、與已知模式不匹配的注入，或通過鉤子未覆蓋渠道到來的注入（例如，注入到依賴項已釋出 README 中、由子代理在瀏覽文件時讀取的內容）。縱深防禦意味著每一層使攻擊更難——而非任何單一層使其不可能。

**漏洞報告。** 請通過 GitHub 私有安全公告提交，地址為 `https://github.com/open-gsd/gsd-core/security/advisories/new`。請勿開啟公開 issue。響應時間表和披露政策請參閱 [SECURITY.md](../../../SECURITY.md)。
