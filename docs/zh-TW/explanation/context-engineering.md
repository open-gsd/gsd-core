# 上下文工程

> GSD Core 的存在原因及其旨在解決的問題。

---

## 問題：上下文腐化

每次 AI 編碼會話都從全新開始。模型讀取你的問題，對其進行推理，然後作出回覆。但一次會話很少只有一輪互動。你會追問後續問題、貼上錯誤資訊、迭代程式碼，並在模型偏離方向時加以糾正。每一輪都會向上下文視窗中新增令牌——這是一個模型在同一時刻能"看到"的有限文本緩衝區。

隨著該視窗逐漸填滿，一些微妙的變化隨之而來。模型不會高調地出錯，它仍然持續作答，但答案的品質會悄然下降。早期的指令被推到它所能關注範圍的邊緣。最初幾輪互動中的細節——你陳述的約束、你達成共識的架構、你標註的邊界情況——都在與後續堆積的內容爭奪注意力。研究人員將這種現象稱為**上下文腐化**。

上下文腐化會以多種方式顯現：

- 模型開始與它此前已認可的決定相矛盾。
- 程式碼風格偏離了會話開始時確立的規範。
- 計劃開始忽略那些明確陳述過、但如今已深埋於歷史記錄中的需求。
- 模型產生幻覺，給出二十條訊息前還正確的檔名或函式簽名。

這些問題都不是模型的 bug。這是 Transformer 注意力機制在長序列上運作的基本屬性。模型並非在"遺忘"——它從未以人類的方式"記住"過任何東西。它在有限視窗中對相關性進行加權，而隨著視窗被積累的噪聲填滿，訊雜比不斷下降。

最直覺的應對方式是使用 `/clear` 重新開始。但這會丟失連貫性。你必須重新解釋背景、重新貼上相關檔案、重新陳述約束條件。會話實質上歸零重啟了。

---

## GSD Core 的答案：全新上下文的子智慧體

GSD Core 的核心洞見是：編碼會話中*大多數*工作根本無需在主上下文中完成。研究、規劃、程式碼編寫和驗證各自是獨立且邊界清晰的任務。每項任務都可以交給一個專門的子智慧體來處理——該智慧體以一個乾淨、精心限定範圍的上下文視窗啟動，並將其結果報告給一個保持精簡狀態的薄層編排器。

這不是上下文腐化的變通之法，而是一種結構性解決方案。

編排器——也就是你的主會話——從不接觸原始檔。它生成智慧體、收集其結果、更新共享狀態，並路由到下一個步驟。正因為它自身承擔的工作很少，其上下文視窗的增長緩慢且可預測。繁重的工作發生在各個智慧體中——每個智慧體都以全新狀態啟動，僅接收完成其任務所需的上下文，並在完成後終止。

考慮這在實踐中意味著什麼。當你執行 `/gsd-plan-phase` 時，編排器會：

1. 載入一個緊湊的 JSON 上下文有效載荷（專案摘要、階段目標、相關配置）。
2. 生成一個擁有 20 萬令牌全新視窗的研究智慧體。
3. 以研究輸出和階段需求為輸入，生成一個規劃智慧體。
4. 生成一個計劃檢查智慧體，在執行前驗證計劃。

每個智慧體都以滿負荷執行，不受會話積累歷史的拖累。當規劃器將其 `PLAN.md` 檔案寫入 `.planning/phases/` 時，該輸出成為持久的產物——而非共享上下文視窗中脆弱的記憶。

## 規格驅動開發與元提示

僅靠上下文工程還不夠。如果一個智慧體以全新狀態啟動，卻接收到模糊的指令，它產生的輸出也將是模糊的。GSD Core 將全新上下文的子智慧體與兩項互補的原則相結合：

**規格驅動開發**意味著每個階段在執行開始之前都會生成結構化產物。`CONTEXT.md` 捕獲來自討論步驟的實現決策。`RESEARCH.md` 記錄研究智慧體的發現。`PLAN.md` 將工作分解為離散的、按依賴關係排序的任務，並附有明確的驗收標準。在執行器智慧體接觸檔案之時，它已擁有一份精確的規格說明——而非對一段漫長對話的重新解讀。

**元提示**意味著智慧體定義本身就是經過精心設計的提示，而非臨時指令。`get-shit-done/workflows/` 和 `agents/` 中的檔案編碼了關於如何限定任務範圍、需要驗證什麼，以及何時上報至人工檢查點的寶貴經驗。使用者無需在每次會話中重新解釋這些知識；它已內嵌於系統自身的提示中。

這種組合是刻意為之的。全新上下文確保每個智慧體清晰推理。規格驅動的產物確保每個智慧體針對*正確的*事物進行推理。元提示確保每個智慧體知道*如何*將其做好。

## `.planning/` 的作用

上下文工程要求知識能在上下文重置後得以保留。GSD Core 為此使用檔案系統。每一項有意義的輸出都以人類可讀的 Markdown 或 JSON 格式寫入 `.planning/`。這意味著：

- 重啟會話（或模型崩潰）不會丟失工作成果。
- 任何後續智慧體都可以直接讀取先前的產物，而無需依賴共享的對話歷史。
- 你可以檢查、編輯規劃產物，或將其提交到 git——它們是純文本，而非資料庫中不透明的狀態。

`STATE.md` 是這個系統的支柱。它記錄專案的當前位置（處於哪個里程碑、哪個階段、哪些計劃已完成）、活躍的決策和阻礙項，以及進度指標。每當工作流啟動時，它都會讀取 `STATE.md` 來定向自身。每當工作流完成一個有意義的步驟時，它都會回寫 `STATE.md`。智慧體不依賴記憶；它們依賴檔案。

## 權衡

這裡需要如實說明權衡之處。

**額外開銷。** 階段迴圈引入了真實的摩擦。將 `/gsd-discuss-phase`、`/gsd-plan-phase` 和 `/gsd-execute-phase` 作為獨立步驟執行，比直接在普通會話中輸入"實現這個功能"需要更多的耗時。對於小型、已充分理解的改動，這種開銷並不值得。

**延遲。** 生成多個擁有全新上下文的子智慧體，比單次上下文內編輯要慢。研究、規劃和執行各自都會產生往返成本。

**簡單任務的繁文縟節。** 如果你只需要重新命名一個變數、修復一個錯別字，或新增一個缺失的匯入，階段迴圈就是殺雞用牛刀。GSD Core 提供了 `/gsd-quick` 和 `/gsd-fast`，用於處理不需要完整階段的臨時工作。請參閱[處理快速任務](../how-to/handle-quick-and-fast-tasks.md)。

當工作足夠複雜、上下文腐化成為真實風險時，階段迴圈才物有所值——例如多檔案功能、橫切重構、跨越數小時或多個會話的工作。其他情況下，請使用更輕量的原語。

一個有用的經驗法則：如果任務可以用一個簡短的提示完整描述，且無需進一步澄清就能在一次智慧體回合中完成，跳過階段迴圈。如果任務需要研究、涉及你近期未讀取的檔案，或依賴尚未確定的決策，階段迴圈則能保護你。
