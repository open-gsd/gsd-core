**The API-coverage seal gate no longer misreads a COVERAGE.md's other tables as capability rows, and accepts sectioned matrices and emphasised decisions.** `parseCoverageMatrix` scanned every pipe-table in the file, so a summary table whose second cell read `INTEGRATE`/`OPT-OUT` was silently pushed as a real capability row (inflating the integrate count and inventing a capability while the blocking gate reported success), a second `| capability | decision | reason |` section errored as an invalid decision, and a bolded or backticked decision was rejected even though the file rendered correctly. The parser now reads rows only from inside a table whose header matches the canonical `Coverage` schema in the shared `TABLE_SCHEMAS` registry (the same seam roadmap/requirements/quick-task/security tables resolve through), strips inline emphasis before validating the decision cell, and treats each canonical header as opening a matrix section so split phases can carry "this phase" and "transferred to a later phase" tables. Fenced code is skipped during the scan, so a canonical-shaped table written as a worked example inside a code fence (as the `api-coverage-plan-pre.md` fragment does) is no longer parsed as real data; and a stray delimiter inside the rows is detected with a strong `-{3,}` match so an all-single-dash data row is rejected precisely instead of truncating the rows that follow it. (#2366)

A coverage row that falls OUTSIDE the table now blocks the seal instead of vanishing. Rows are read only from inside a well-formed table, and every later piped line was skipped silently — so a matrix split by a blank line, a prose line, or an HTML comment lost every row after the split with zero errors while the seal still reported `valid: true` (the reporter's case dropped two `OPT-OUT` rows, the decisions the gate exists to force). The parse walk now distinguishes a row inside another table from a row inside no table at all: rows of a summary or threat table stay correctly ignored, while a coverage-shaped row belonging to no table is a loud error naming it — including a row wider than three columns, which is usually wide because an unescaped `|` split it. The one-contiguous-block requirement, and repeating the header to open a new matrix section, are documented in `gsd-core/references/api-coverage.md` and the `api-coverage-plan-pre.md` fragment. (#2366)

Diagnostics on the same path are sharpened: a table headed like the matrix but not matching the canonical schema (an extra column, or a reordered one) is named instead of surfacing as "matrix is empty"; an unterminated code fence is reported with its line number rather than silently masking every row below it; a row's width is compared to its own header instead of a hardcoded 3, so registering a second `Coverage` variant cannot silently reject every row; the reader resolves the header through `matchTableSchema` (which gains a `caseInsensitive` option) rather than walking `TABLE_SCHEMAS` itself, closing a divergence where the two disagreed about `| Capability | Decision | Reason |`; and a correctly GFM-escaped `\|` in a reason is accepted and re-escaped on render instead of being rejected as an unsupported pipe. (#2366)
