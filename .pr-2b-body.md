## Feature summary

Graduates the **#853 orchestrator-backgrounding decision** from a scattered `RUNTIME === 'codex'` workflow-prose check to a **typed, documentation-sourced** engine decision. The plan/execute workflows now call a `gsd_run query dispatch-should-flatten` whose answer is computed from a new documentation-sourced `backgroundDispatch` dispatch sub-axis on each runtime descriptor — eliminating the runtime-name residue for this decision.

> ⚠️ **Behavior change (approved):** cursor is now **background-eligible** (it inlined before). Cursor's docs (Cursor 2.5) document that backgrounded subagents retain nested dispatch, so the data-driven rule correctly enables it — the same generalization #853 always implied, now applied to *any* host whose docs say it's safe, not just the one we hardcoded. codex is unchanged; the other 14 hosts inline (documented-false or undocumented→fail-closed).

## Implementation notes

The #853 safety contract is "never background an orchestrator that can't itself spawn sub-agents" — so the decision **fail-closes to inline** at every exit: unknown runtime / missing dispatch / registry error / null input all return `true`. The discriminator (`backgroundDispatch` = "can a backgrounded sub-agent spawn further named sub-agents?") was sourced per host — e.g. Claude's docs explicitly say *"Background subagents... cannot spawn further"* (→ `false`), codex's docs say sub-agents retain spawn capability (→ `true`). Hardened across Codex adversarial (no fail-open), security (clean), and code review: the Compound Action preamble that still said "On Codex" was caught and converted; `shouldFlattenDispatch` made null-safe; `backgroundDispatch` made a required field.
